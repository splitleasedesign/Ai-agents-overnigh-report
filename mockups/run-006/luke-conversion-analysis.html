A/B Test Roadmap

Prioritized experiments with full specifications, success criteria, and implementation details.

TEST-001 P0 - Critical

Sticky Sign CTA Placement

If we show a persistent sticky "Sign Now" CTA at the bottom of the document viewer, then signature completion rate will increase by 20% because users won't abandon due to scroll fatigue.

Control
Sign button only after scroll
Variant
Sticky CTA visible throughout
Duration
2 weeks or 200 signatures
Segment
All users with unsigned leases
Primary Metric Signature completion rate
Success Threshold >15% lift (95% confidence)
Guardrail No increase in support tickets

Implementation Notes

  • CTA should include "I've read and agree" checkbox in sticky bar
  • Progress indicator showing scroll position in document
  • CTA remains disabled until 80% scroll depth OR 2 minutes elapsed
  • Mobile: Full-width sticky bar at bottom, 56px height

Secondary Metrics

  • Time from document view to signature
  • Scroll depth before signing
  • Mobile vs Desktop completion rate delta

Rollout Plan

Phase 1: 10% traffic for 3 days (smoke test). Phase 2: 50% traffic for full duration. Phase 3: 100% rollout if success criteria met.

TEST-002 P0 - Critical

One-Click Payment

If we allow users with saved payment methods to pay in one click, then payment completion rate will increase by 25% because reduced friction removes abandonment opportunities.

Control
3-step: View > Review > Confirm
Variant
"Pay Now with ****4242" button
Duration
1 billing cycle (30 days)
Segment
Users with saved payment method
Primary Metric Payment completion rate
Success Threshold >20% lift, no chargebacks
Guardrail Payment errors, disputes

Implementation Notes

  • Include 30-second "undo" period after one-click pay
  • Show payment method icon (Visa, Mastercard, etc.) in button
  • Confirmation toast with receipt link
  • Fallback to full flow if payment method expired

Secondary Metrics

  • Time from invoice view to payment complete
  • Undo rate (should be <2%)
  • Support ticket volume

Risk Mitigation

If undo rate exceeds 5%, auto-pause experiment and add review step back.

TEST-003 P1 - High

DCT Response Email Buttons

If we include Accept/Decline buttons directly in DCT notification emails, then response rate within 24 hours will increase by 30% because removing the login step reduces friction.

Control
"View Request" link > Login > Decide
Variant
Accept/Decline buttons in email
Duration
2 weeks or 50 DCT requests
Segment
All DCT recipients (Hosts)
Primary Metric Response rate within 24h
Success Threshold >25% lift
Guardrail Accidental decline rate <5%

Implementation Notes

  • Use secure tokenized links for each action button
  • Include "View Details" link for users who want more info
  • Show request summary inline: dates, amount delta
  • Confirmation page after email action with undo option

Secondary Metrics

  • Response rate within 4 hours (urgency signal)
  • Users who undo within 1 hour
  • Counter-proposal rate (may decrease if Accept is too easy)
TEST-004 P1 - High

Mobile Week Strip vs Month View

If we replace the cramped month calendar with a swipeable week strip on mobile, then DCT completion rate on mobile will increase by 35% because touch targets meet 44px minimum and cognitive load is reduced.

Control
Scaled-down month grid
Variant
Swipeable week strip + full-screen picker
Duration
2 weeks or 100 mobile DCT actions
Segment
Mobile users only (<768px)
Primary Metric DCT completion rate (mobile)
Success Threshold >30% lift
Guardrail Task time not >2x longer

Implementation Notes

  • Week strip: 7 cells at 53px each = 371px (fits 375px viewport)
  • Swipe left/right to navigate weeks
  • Tap any date to open full-screen calendar picker
  • Selection state persists at bottom of sheet

Based on Jony Ive's Analysis

Month grid at 375px = 40px touch targets (below 44px minimum). Week strip solves this while maintaining calendar mental model.

TEST-005 P1 - High

"Request More Time" Option

If we add a "Request More Time" button for DCT responses, then the expiration rate will decrease by 40% because procrastinators will engage rather than avoid.

Control
Only Accept/Decline/Counter
Variant
Add "Request More Time" (+24h, one-time)
Duration
3 weeks or 75 DCT expirations
Segment
All DCT responders
Primary Metric DCT expiration rate
Success Threshold >35% reduction in expirations
Guardrail Overall response time not >2x

Implementation Notes

  • One-time use per request (cannot extend again)
  • Requester notified: "[Name] needs more time. New deadline: [date]"
  • Extension button disabled after use, shows "Extended until [time]"
  • Track: users who extend vs users who let expire

Behavioral Insight

Per Ariely's research on procrastination, giving an explicit "I need more time" action surfaces users who would otherwise ghost. This converts non-responses into continued engagement.

Dark Pattern Prevention Checklist

All designs verified against Dieter Rams' dark pattern prevention checklist:

  • ✓

    No forced scroll-to-bottom before Sign button

    Sticky CTA pattern respects user time. Scroll tracking is for analytics, not gating.

  • ✓

    "Request Changes" has equal visual weight to "Sign"

    Both buttons are same size, no color manipulation to favor signing.

  • ✓

    No red styling on "Due" status before actually late

    Due = Amber, Past Due = Red. Colors accurately reflect state.

  • ✓

    No "Avoid late fees!" when there are no late fees

    Copy only mentions fees when they genuinely apply.

  • ✓

    No large anxiety-inducing countdown timer for DCT

    Human-readable "23 hours left" not "00:23:47:12".

  • ✓

    Original dates confirmed as safe fallback

    Expired DCT copy: "Your original dates remain confirmed".

Final Recommendations

The Leases Management Pages represent Split Lease's highest-stakes conversion surface. Three actions determine revenue and retention:

  1. Sign Lease - Without signature, no legal relationship exists
  2. Pay Invoice - Revenue depends on frictionless payment
  3. Respond to DCT - Flexibility is the product differentiator

The team's designs are conversion-aware. Josef's component system, Dieter's trust patterns, Milton's copy framework, and Jony's mobile adaptations all support conversion goals.

The critical gap is instrumentation. Without event tracking, we cannot measure baselines, run experiments, or validate improvements. This is the single most important P0 item.
